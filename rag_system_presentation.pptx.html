Core Domain Concepts: Agent & API Keys

Agent Concept

An Agent represents one business-specific chatbot with strict isolation at the retrieval layer.

Agent Properties

Strict Role Definition

System prompt defines agent behavior and boundaries

Specified Ollama Chat Model

Configurable LLM per agent (Llama 3.1, Qwen 2.5, etc.)

Specified Embedding Model

High-quality embeddings (nomic-embed-text, bge-m3, etc.)

Isolated RAG Knowledge Base

Dedicated vector space in Qdrant per agent

Multiple API Keys

One-to-many relationship between agent and keys

Configurable Chat History

Customizable conversation memory length

API Key Management

Secure authentication with complete isolation between agents.

Key Properties

Secure Hashes Only

One-way hashing (bcrypt/argon2)

No Expiration

Keys remain valid until explicitly revoked

Soft Delete Revocation

Revoked keys are marked, not removed

Request Rejection

Revoked keys are immediately blocked

Isolation Guarantee

Agents are completely isolated at the retrieval layer. One agent's documents and vectors are never accessible to another agent, ensuring data privacy and security in multi-tenant deployments.

{

"file_id": "<ObjectId>",
"chunk_index": <number>,

Filtering Capabilities

Filter by file_id

Implementation Notes

Collections are created dynamically when an agent is created. Vector dimensions must match the agent's embedding model. Payload indices should be created on file_id and section for optimal performance.

Reranking & Cross-Encoder Models

Critical Reranking Step

After retrieval, the top 10-20 candidates are reranked using cross-encoders to improve precision. This is a mandatory step for production quality.

Why Reranking Matters

Higher Precision

Cross-encoders deeply analyze query-chunk relevance

Quality Control

Only high-confidence chunks reach the LLM

Hallucination Prevention

Reduces irrelevant context that causes hallucinations

Approved Rerankers

bge-reranker

Balanced performance

ms-marco

Proven benchmark

colbert

Late interaction

Quality Threshold

Only high-confidence chunks may reach the LLM. Low-relevance results are filtered out.

Score ≥ θ

θ = minimum relevance threshold

Reranking Process

1

Take top 10-20 retrieval candidates

2

Score each with cross-encoder

3

Reorder by relevance score

4

Filter by minimum threshold

5

Pass only high-confidence chunks to LLM

Thresholding, Safety & Context Assembly

Mandatory Safety Mechanisms

The system must discard low-similarity results to prevent hallucination. If no confident context exists, the system fails safely.

Safe Failure Response

"The information is not available in the provided knowledge base."

Context Assembly Rules

Structured & Labeled: Each chunk includes source attribution

Deduplication: Redundant chunks are removed

Size Enforcement: Maximum context size respected

RAG PIPELINE IMPLEMENTATION

File Deletion & RAG Integrity

Three-Step Deletion Process

When a file is deleted, the system follows a strict three-step process to ensure zero orphan vectors and maintain RAG integrity.

1

Mark as Deleted

File metadata is marked as deleted in MongoDB for audit trail and compliance

is_deleted: true

2

Remove Vectors

All vectors are removed from Qdrant using file_id filter

filter: { file_id: $id }

3

Preserve Metadata

File metadata is retained for audit, compliance, and historical tracking

retained: forever

Benefits of This Approach

No Orphan Vectors

Complete cleanup prevents vector space pollution

Audit Trail

Metadata preserved for compliance

Recoverable

Soft delete enables restoration if needed

Zero Orphan Vector Policy

Critical Requirement: No vectors may exist in Qdrant without corresponding metadata in MongoDB. This ensures consistency and prevents ghost retrievals.

Regular audits verify integrity between MongoDB and Qdrant
